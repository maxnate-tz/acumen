# MAXNATE Acumen — Design Plan

**Product:** MAXNATE Acumen — *Sharp insight for AI agent workflows.*
**Purpose:** A **generation system** that studies any project, detects gaps, and automatically produces project-specific OpenCode/Claude Code skills, commands, checklists, state files, and MCP suggestions — so no developer ever needs to manually structure their AI-assisted workflow again.

**Source projects studied:**
- **Dr Adelaida** — Nuxt 3 monolithic CMS (46 modules, 13 commands, 54% project-specific)
- **Bisou Bailey Dispensary** — Express + Nuxt 4 monorepo (8 commands, 63% project-specific)
- **TRHM Main Website** — Nuxt 4 CMS rebrand (4 commands, 100% project-specific)
- **Mwanza** — Extracted from Bisou Bailey (12 commands, 67% project-specific)

**Core truth from references:** 54-100% of skills/commands across all four projects are project-specific. A static catalog shipping 6 fixed commands solves 0% of the real problem. This repository is a **generator**, not a catalog.

## Part 1: Design Pillars

### Pillar 1: Generation System, Not Static Catalog

```
┌──────────────────────────────────────────────────┐
│           CONTINUOUS GENERATION LOOP             │
├──────────────────────────────────────────────────┤
│                                                   │
│  Bootstrap                                        │
│    ↓                                              │
│  STUDY → ProjectModel (framework, modules, MCPs)  │
│    ↓                                              │
│  COMPARE → GapAnalysis (missing patterns, tools)  │
│    ↓                                              │
│  GENERATE → Tailored commands + skills + state    │
│    ↓                                              │
│  OBSERVE → Developer adds modules, new patterns   │
│    ↓                                              │
│  SUGGEST → "I notice 3 new blog endpoints.        │
│             Generate a blog-audit command?"        │
│    ↓                                              │
│  User confirms → GENERATE again → LOOP            │
│                                                   │
└──────────────────────────────────────────────────┘
```

### Pillar 2: MCP-First Task Routing

Every command phase scans connected MCPs before executing:

```
RECEIVE TASK → SCAN .mcp.json → MAP TASK → MCP TABLE
IF MCP AVAILABLE → USE IT (HARD GATE)
IF UNAVAILABLE → FALLBACK MANUALLY (score capped)
```

Degradation tiers:

| Tier | Condition | Behavior | Max Score |
|---|---|---|---|
| FULL | All relevant MCPs connected | Optimal tool every step | 100 |
| PARTIAL | Some connected | Use available, fallback for rest, cap | varies |
| NONE | Zero connected | All manual, `[MCP-DEGRADED]` flag | 75 |

### Pillar 3: Dual-Stack Single Source of Truth

```
project/
├── .claude/              ← ALL logic lives here
│   ├── skills/           ← PRIMARY format: directory per skill
│   │   └── audit-module/
│   │       ├── SKILL.md          ← Router + YAML frontmatter (loaded always)
│   │       ├── workflows/        ← Phase-by-phase procedures (loaded on demand)
│   │       └── references/       ← Anti-pattern catalog, scoring (read when needed)
│   ├── commands/         ← LEGACY format (still supported, auto-detected)
│   ├── checklists/       ← Audit reports, module catalog
│   └── state/            ← JSON state files
├── .opencode/            ← THIN WRAPPERS (delegate to .claude/)
│   ├── skills/<name>/SKILL.md  ← "Read .claude/skills/<name>.md"
│   └── commands/<name>.md      ← "Read .claude/commands/<name>.md"
├── AGENTS.md             ← Universal AI agent guide
└── CLAUDE.md             ← Claude Code-specific instructions
```

The directory skill format enables **progressive disclosure**: SKILL.md is the router (~200 tokens), workflows/ holds phase details, references/ holds domain knowledge. This reduces context cost 60-80% vs loading everything into one file (confirmed by Anthropic's 2026 Skills unified format). The flat `.md` format is the v1 fallback and is auto-detected from the project's existing structure.

Two wrapper formats supported (bootstrap detects which to use):

| Format | Used By | Structure |
|---|---|---|
| Flat commands | Dr Adelaida, Mwanza | `opencode/commands/<name>.md` |
| Directory skills | Dr Adelaida, Mwanza | `opencode/skills/<name>/SKILL.md` |

### Pillar 4: Score-Driven Quality

Every module gets a weighted score. Formula adapts per domain:

```
SCORE = sum(section_weight × tier_points_earned / tier_points_total)

TIERS:
  Tier 1 (MUST):  2.0 pts per check — CRITICAL severity
  Tier 2 (SHOULD): 1.0 pts per check — MAJOR severity
  Tier 3 (MAY):    0.5 pts per check — MINOR severity

SECTION WEIGHTS (configurable):
  Internal Conventions → ×2.0
  Cross-Module → ×2.0
  Data Integrity → ×1.0 (×2.0 for healthcare/fintech)
  Code Quality → ×1.0
  Industry Standards → ×2.0

CAPS:
  BLOCKER → max 50
  CRITICAL → max 70
  HIGH → max 85
  Missing sequential-thinking → max 60
  Missing WebSearch → max 70
  Missing playwright (frontend) → max 85
```

### Pillar 5: Skill/Command Lifecycle

Skills and commands are living artifacts with a defined lifecycle:

```
CREATION triggered by:
  - New module detected (study phase)
  - New pattern found (cross-module analysis)
  - Developer request
  - Gap analysis output

DEPRECATION via frontmatter:
  ---
  status: deprecated
  replacedBy: audit-module
  deprecationReason: Merged into unified audit system
  ---

DRIFT DETECTION:
  - Compare command spec file:line references against actual module state
  - If >20% of references stale → flag as DRIFTED, suggest regeneration
  - Drift is documented in a machine-readable drift catalog

MERGE:
  - When two skills converge on same pattern
  - Auto-suggest merge with diff preview
```

## Part 1a: Implementation Scope — v1 / v2 / v3

The reference projects (Dr Adelaida, Bisou Bailey, TRHM, Mwanza) have zero MCP servers, zero watchers, zero pipeline compositors, zero continuous observation — and they are highly effective. The spec's core innovation is "study project → detect gaps → generate tailored artifacts." Everything beyond v1 is optimization for problems that don't exist yet.

| Component | Scope | Reason |
|---|---|---|
| npm CLI: `bootstrap [--update\|--force]` | **v1** | Core value — study project, generate all artifacts |
| Single .md skill output format | **v1** | Proven pattern from all 4 reference projects |
| 6 core templates | **v1** | AGENTS.md, CLAUDE.md, audit-module, arch-lint, deploy-check, module-registry |
| Nuxt 3 + Express 5 detectors | **v1** | Matches reference projects |
| CMS + Healthcare domains | **v1** | Matches reference project domains |
| MCP scanning + suggestion reports | **v1** | Read .mcp.json, suggest missing MCPs with domain-appropriate caps |
| Generated `.opencode/` thin wrappers | **v1** | Two wrapper formats (flat .md + directory/SKILL.md) |
| `--update` / `--force` / `--dry-run` flags | **v1** | Essential for re-running on bootstrapped projects |
| | | |
| MCP Server (Streamable HTTP, daemon) | **v2** | Continuous observation requires persistent process |
| Directory skill format (SKILL.md + workflows/ + references/) | **v2** | Token-budget optimization; single .md works for v1 |
| `generate` and `check-drift` CLI commands | **v2** | Bootstrap handles 80%+ of use cases |
| Filesystem watchers (notification-driven) | **v2** | Only with MCP server; MCP can't auto-execute tools |
| Anti-pattern auto-detection (basic) | **v2** | Needs adoption data to calibrate |
| Framework: Next.js | **v2** | Expand beyond Nuxt + Express |
| | | |
| Pipeline compositor + MCPC export bridge | **v3** | 5 MCP protocol limitations make this harder than assumed |
| Anti-pattern auto-detection (full cross-module) | **v3** | Needs adoption at scale |
| Frameworks: Django, Spring Boot | **v3** | Build when users need them |
| Domains: ecommerce, fintech, saas | **v3** | Content work; no architecture change |
| Brownfield extraction workflow | **v3** | Niche pattern; exists in Mwanza reference |

The MCP specialist agent found 5 protocol limitations that push the pipeline compositor to v3: (1) MCP tool timeouts of 30-300s — audit pipelines exceed this, (2) no server-to-server tool discovery, (3) `resources/subscribe` API being replaced, (4) modifying `.mcp.json` requires client restart, (5) stdio kills server on session end.

### Bootstrap Modes

`/bootstrap` supports three modes for re-running on an already-bootstrapped project:

| Mode | Flag | Behavior |
|---|---|---|
| **Fresh** | (no flag) | Generate from scratch — fail if `.claude/` already exists |
| **Update** | `--update` | Re-study, detect new gaps, suggest additions only. Never overwrite existing custom commands. Merge missing state files, add new skills. |
| **Force** | `--force` | Regenerate EVERYTHING — overwrites existing `.claude/` and `.opencode/`. **WARNING:** Custom work will be lost. Print diff of what changed before applying. |

Update mode is the safest for ongoing use. It works like a package manager upgrade — add what's new, keep what exists, flag conflicts.

## Part 2: The Generation System (Core Innovation)

### 2.1 Project Study Phase → ProjectModel

Before generating anything, the system produces a comprehensive ProjectModel:

```typescript
interface ProjectModel {
  // Identity
  framework: "nuxt3" | "nextjs" | "express" | "django" | "spring-boot" | "unknown"
  language: "typescript" | "javascript" | "python" | "java" | "unknown"
  monorepo: boolean
  workspaceStructure?: "turborepo" | "nx" | "pnpm-workspace" | "none"

  // Data layer
  orm: "prisma" | "drizzle" | "sequelize" | "typeorm" | "sqlalchemy" | "none"
  db: "postgresql" | "mysql" | "sqlite" | "mongodb" | "unknown"

  // Auth
  auth: "next-auth" | "lucia" | "passport" | "custom-jwt" | "django-auth" | "none"

  // Testing
  testRunner: "vitest" | "jest" | "pytest" | "rspec" | "none"
  e2eFramework: "playwright" | "cypress" | "selenium" | "none"

  // Package management
  packageManager: "npm" | "pnpm" | "yarn" | "unknown"

  // Modules (auto-classified)
  modules: Array<{
    name: string
    path: string
    type: "foundation" | "infrastructure" | "core-business" | "content" | "frontend" | "unknown"
    dependencies: string[]
    files: string[]
    endpoints: string[]
    hasVerifier: boolean
    hasAudit: boolean
    score: number | null
  }>

  // Existing artifacts
  existingCommands: string[]
  existingSkills: string[]
  antiPatterns: number
  mcps: string[]
  domains: string[]          // inferred healthcare, ecommerce, etc.
  projectType: string        // full-stack, api, mobile, cms, etc.

  // Existing plan phases (from build-state.json / MODULES.md / phase dirs)
  planPhases: Array<{
    number: number
    name: string
    description: string
    status: "pending" | "in-progress" | "completed"
    modulesInPhase: string[]
  }>

  // Infrastructure
  hasDocker: boolean
  hasCI: boolean
  hasStateFiles: boolean
  hasVerifierScripts: boolean
}
```

**Study pipeline:**
1. Scan directory structure (guess framework from config files, layout)
2. Read `package.json` / `pyproject.toml` / `Cargo.toml` etc.
3. Detect ORM from dependencies
4. Detect auth from imports/handlers
5. Scan for `.mcp.json` and list connected servers
6. Walk modules directory → classify each by location and content
7. Detect existing `.claude/` artifacts
8. Detect CI/CD configs, Docker files
9. Extract plan phases from `build-state.json`, `MODULES.md`, phase-named directories, or task files
10. Produce ProjectModel

### 2.2 Gap Analysis → Prioritized Recommendations

Compare ProjectModel against a pattern library and produce:

```typescript
interface GapReport {
  gaps: Array<{
    pattern: string           // "audit-module" | "arch-lint" | "phase-verifier" | etc.
    severity: "critical" | "high" | "medium" | "low"
    impact: string            // "No systematic quality enforcement"
    effort: "high" | "medium" | "low"
    scoreImpact: number       // Current score lost without this
    autoGeneratable: boolean  // Can the generator produce it?
    dependsOn: string[]       // Prerequisites
  }>
  recommendations: Array<{
    command: string           // "arch-lint"
    reason: string            // "8 architecture checks exist but no /arch-lint wrapper"
    template: string          // Which template to use
    effort: string
    estimatedScoreGain: number
  }>
}
```

**Pattern library (built-in, extendable):**

| Pattern | Detect When | Severity If Missing |
|---|---|---|
| `audit-module` | Multiple modules detected | CRITICAL |
| `fix-module` | audit-module exists | CRITICAL |
| `deploy-check` | CI/CD detected | HIGH |
| `arch-lint` | Architecture conventions exist | HIGH |
| `anti-pattern-catalog` | Any module code exists | HIGH |
| `build-orchestrator` | Multi-phase project | MEDIUM |
| `module-registry` | 5+ modules detected | MEDIUM |
| `phase-verifier` | Backend modules detected | HIGH |
| `audit-lock` | Multiple developers | MEDIUM |
| `live-baseline` | Production deployed | MEDIUM |
| `mockup-audit` | mockup/ directory exists | HIGH |
| `spec-as-truth` | Upstream contract detected | HIGH |
| `drift-catalog` | Generated artifacts exist | MEDIUM |
| `brownfield-extraction` | Extraction scripts needed | LOW |

**Pattern file schema (`.pattern.md`):**

Every `.pattern.md` file in the pattern library uses this format:

```markdown
---
pattern: audit-module
severity: critical                    # critical | high | medium | low
detection:                            # Conditions that trigger this pattern
  hasModules: true                    # Must have identified modules
  minModuleCount: 1
  hasAntiPatternCatalog: preferred
  hasCommands: []                     # Existing commands to check
  hasDirs: []                         # Directories to check existence
  hasFiles: []                        # Files to check existence
dependsOn: [anti-pattern-catalog]     # Patterns that must be generated first
references: [score-engine]            # Core specs to consult during generation (optional)
generates:                            # Files this pattern produces
  - .claude/commands/audit-module.md
  - .opencode/commands/audit-module.md
  - .claude/checklists/<module>-production-readiness.md
templates:                            # Templates to render
  - commands/audit-module.md.template
stateFiles:                           # State schemas to create
  - audit-state.json.template
  - module-registry.json.template
---
```

This schema drives the gap analyzer: it reads `detection.hasModules` + `detection.minModuleCount`, checks the ProjectModel, and includes/excludes the pattern from the gap report. The gap analyzer never hardcodes detection logic — it reads it from the pattern file.

### 2.3 Command/Skill Generator → Tailored Output

Given a gap and ProjectModel, produces complete `.claude/` and executable artifacts:

**Two generators, two outputs:**
- `command-generator.ts` → `.claude/commands/<name>.md` + `.opencode/<name>.md` (markdown instructions)
- `verifier-generator.ts` (v2) → `backend/scripts/verify-phaseN-<name>.mjs` + shared lib + npm script (executable code)

Both read from the same pattern file and template, but produce different artifact types.

```
INPUT:
  - Command purpose: "arch-lint"
  - ProjectModel (Nuxt 3, Prisma, custom auth, Vue composables)
  - Available MCPs: [playwright, sequential-thinking, github]
  - Existing conventions: anti-pattern-catalog.md, AGENTS.md rules

OUTPUT:
  .claude/commands/arch-lint.md with:
    - Framework-adapted checks (not Express grep → scan server/api/)
    - Project-specific anti-pattern references
    - Correct MCP routing for available MCPs
    - Hard gates matching project conventions
    - Project-appropriate score caps

   .opencode/commands/arch-lint.md thin wrapper

   .claude/checklists/ARCH-LINT-TEMPLATE.md
```

**Template variable resolution:**

| Placeholder | Resolution |
|---|---|
| `{{FRAMEWORK}}` | From ProjectModel |
| `{{MODULE_PATHS}}` | From ProjectModel.module scan |
| `{{ANTI_PATTERN_COUNT}}` | From anti-pattern catalog |
| `{{MCP_ROUTING}}` | Auto-generated from connected MCPs |
| `{{CHECK_COMMANDS}}` | Framework-adapted grep patterns |
| `{{HARD_GATES}}` | From project convention template |
| `{{DOMAIN_SPECIFIC}}` | From inferred domain |

**`/create-command` meta-command (two modes):**

```markdown
# /create-command

Usage: /create-command <purpose> [--interactive]
       /create-command arch-lint             (automatic — resolve everything from ProjectModel)
       /create-command blog-audit --interactive  (ask developer for each decision)

## Automatic mode (default)
1. Re-read ProjectModel
2. Find matching pattern in patterns/ library (by purpose name)
3. Read the pattern's .pattern.md → get template name + detection criteria
4. Read the template → resolve all {{PLACEHOLDERS}} from ProjectModel
5. Generate: .claude/commands/<name>.md + .opencode/<name>.md
6. Report what was generated with diff preview
7. Ask: "Apply? [y/N]"

## Interactive mode (--interactive)
1. Re-read ProjectModel
2. List available patterns from patterns/ library
3. Show: "Pattern arch-lint needs: {{FRAMEWORK_CHECKS}}, {{CHECK_COUNT}}"
4. Ask: "Which checks to include? [all / select / custom]"
5. Ask: "Which MCPs to require? [detected: playwright, sequential-thinking]"
6. Ask: "Score cap on failure? [default: 85]"
7. Generate with selected options
8. Report with diff preview
9. Ask: "Apply? [y/N]"

After generation:
- Add to command registry
- Run gap analysis again (new command may reveal new gaps)
```

### 2.4 Continuous Suggestion Loop

The system doesn't just run once. It observes and suggests:

```
Bootstrap → Study → Generate
     ↓
Developer uses commands → New modules added
     ↓
System detects: "3 new API endpoints in blog/ module"
     ↓
Suggestion: "You've added blog CRUD endpoints.
             Would you like me to generate a blog-audit command?"
     ↓
Developer: "Yes" → System generates blog-audit.md
     ↓
System updates: MODULES.md, module-registry.json, state files
     ↓
Continues observing...
```

**Suggestion delivery mechanism:**

In v1, suggestions are printed to stdout by the CLI. In v2 (MCP server), suggestions use **MCP prompts** — the server constructs dynamic prompt templates, sends `notifications/prompts/list_changed`, and the client surfaces them as clickable conversation starters. Prompts (not tools) are the correct primitive for proactive suggestions: tools require the client to poll or the user to trigger; prompts use server-initiated notifications.

**Suggestion triggers:**

| Trigger | Condition | v1 Behavior | v2 Behavior |
|---|---|---|---|
| New module detected | 3+ files in a new directory | Print to stdout | Dynamic MCP prompt: "Audit 3 new blog endpoints?" |
| Recurring bug | Same pattern across 3 modules | Print to stdout | Dynamic MCP prompt: "Anti-pattern across 3 modules?" |
| MCP added | New MCP in `.mcp.json` | Print to stdout | Regenerate MCP routing in all commands |
| MCP removed | MCP removed from `.mcp.json` | Print to stdout | Playwright removed — 3 commands degraded |
| Missing state file | No `build-state.json` | Print to stdout | Generate state file |
| Template drift | >20% stale references | Print to stdout | Regenerate affected commands |
| New pattern discovered | Manual trigger | Run `/create-command` | Dynamic MCP prompt |

### 2.5 Anti-Pattern Auto-Detection

Monitor for recurring bugs and auto-expand the catalog:

```
1. Track: grep across all module API files for known bad patterns
2. Detect: same class of bug in 3+ different modules
3. New pattern candidate found:
   - Pattern: "Missing idempotency key on POST"
   - Files: payments/post, orders/post, appointments/post
   - Detection: grep -rn "idempoten"
4. Generate anti-pattern entry:
   - Name + severity
   - Detection grep command
   - Fix template from first occurrence
   - Add to catalog
   - Update pre-audit scan script
5. Report: "New anti-pattern detected across 3 modules"
```

## Part 3: Universal Framework (Default Shipments)

**Industry standards alignment (2026):** Anthropic unified Commands into Skills in 2026. The recommended skill format uses YAML frontmatter with at minimum: `name`, `description`, `disable-model-invocation`, `user-invocable`, `allowed-tools`. The `.claude/commands/` directory still works (legacy support) but `.claude/skills/<name>/SKILL.md` is the recommended format. Generated skills should include this frontmatter for invocation control and safety. The directory format with progressive disclosure (SKILL.md router → workflows/ → references/) aligns with the unified Skills specification and reduces context cost 60-80% for complex skills.

The repo ships these as STARTING POINTS — they are overridden/extended by generation:

### 3.1 Universal Skills

| Skill | Purpose | Always Generate? |
|---|---|---|
| `code-audit.md` | Generic audit methodology (6 phases) | YES — base template |
| `code-fix.md` | Generic fix-reaudit loop (11 phases) | YES — base template |
| `deploy-gate.md` | Pre-deploy verification skeleton | YES — if CI detected |
| `api-pattern.md` | Framework-adaptive API development | YES — adapted per framework |
| `anti-patterns.md` | Cross-language bug patterns | YES — stack-adapted |

### 4.5 MCP Protocol Limitations for Pipeline Execution

These protocol realities were discovered during architecture review and constrain v2/v3 design:

1. **No server-initiated tool execution.** MCP servers cannot autonomously run tools. They can only respond to client requests. Watchers can update resources/prompts and send notifications, but a human or Claude must decide to act. "Continuous observation" is limited to suggestion, not autonomous execution.

2. **Tool timeouts (30-300 seconds).** Full audit pipelines with web-search + sequential-thinking + Playwright across multiple phases exceed standard MCP tool timeouts. Long-running pipelines need progress notifications or polling-based decomposition.

3. **No server-to-server tool discovery.** An MCP server cannot enumerate another server's tools via the protocol. It must read `.mcp.json` directly from the filesystem or receive the capability map as a tool argument from the client.

4. **Client restart on config change.** Writing to `.mcp.json` (e.g., to suggest adding a new MCP server) requires the client to restart before the new server is available. There is a UX gap between suggestion and availability.

5. **`resources/subscribe` API being replaced.** The next MCP spec revision (post-2025-11-25) replaces subscribe/unsubscribe with a `subscriptions/listen` stream. Any watcher integration should target the new API.

6. **stdio kills the server on session end.** When Claude Code exits, stdio-based MCP servers die. Persistent observation requires Streamable HTTP transport with daemon management (systemd, Docker, or VS Code extension). This is a v2 concern but constrains the v2 architecture.

These are not blockers for v1 (which doesn't use MCP as a runtime), but they explain why the pipeline compositor is deferred to v3 and constrain the v2 MCP server design.

## Part 5: Complete Extracted Patterns (All 16)

### Pattern 1: Module Audit System (Dr Adelaida)

**6 phases:** Phase 0 (anti-pattern scan) → Phase 1 (industry research) → Pass 1 (deep trace) → Pause 1 (sequential-thinking) → Pass 2 (gap hunt) → Pause 2 (sequential-thinking) → Pass 3 (final verification) → Report

**Hard gates:** All phases mandatory. Skip = void, cap 40. Every finding needs MCP verification.

### Pattern 2: Fix-Reaudit Loop (Dr Adelaida)

**11 phases:** Plan → Fix CRITICAL → Fix HIGH → Fix MEDIUM+LOW → Full Re-Audit → UI Verify → Commit → Push → Deploy Gate → Staging Deploy + Live Test → Sign-Off

**Rules:** Full re-audit after every phase. Never advance below 90. One commit per gap. Sequential-thinking before every fix.

### Pattern 3: Anti-Pattern Catalog (Dr Adelaida + Bisou Bailey)

**Structure:** Named pattern → severity → description → detection grep → real example (file:line) → fix template (before/after) → when to check

**Auto-expansion:** When same class of bug detected across 3+ modules, system auto-generates new catalog entry.

### Pattern 4: Module Registry + State Tracking (Bisou Bailey)

JSON catalog of all modules with: path, status, files, routes, verifier, system-core boundary.

**Domain-specific state files:** `blog-handling-state.json`, `image-handling-state.json`, `trhm-build-state.json`. Generated per-domain by the bootstrap system.

### Pattern 5: Phase Verifier Pattern (Bisou Bailey)

Every backend feature has a Node.js verification script that:
1. Creates a fresh disposable database (`<prefix>_<random>`)
2. Runs isolated from any existing state
3. Tests happy path AND error paths
4. Is included in `verify:local` sweep
5. Tears down everything on completion (server + DB)

**Verifier library provides:** `createVerifyContext()` → `createDatabase()`, `dropDatabase()`, `migrateDatabase()`, `startServer()`, `waitForHealth()`, `requestJson()`, `cleanup()`

### Pattern 6: Architecture Lint (Bisou Bailey /arch-lint)

8 grep-based checks: system-core boundary → route registration → composable compliance → hardcoded URLs → god files → validation coverage → shared contract purity → migration integrity

## Part 8: Key Design Decisions

1. **Generation over catalog.** The repo ships templates + a generator engine, not a fixed set of commands. 54-100% of commands in reference projects are project-specific — a catalog can't solve that.

2. **Study before generate.** Nothing is generated without a ProjectModel. The study phase is the prerequisite for everything.

3. **Gaps drive generation.** The system doesn't generate everything. It compares what exists against known patterns and generates ONLY what's missing.

4. **Continuous loop, not one-shot.** Bootstrap → study → generate → observe → suggest → generate again. The system evolves with the project.

5. **MCP-first, not MCP-only.** Every phase checks for MCP availability. Use if present. Fallback gracefully if not. Cap scores accordingly.

6. **Domain-weighted scoring.** Caps are not absolute — they're relative to the domain. Healthcare without web-search is worse than CMS without it.

7. **Drift is inevitable, document it.** Generated artifacts will drift from reality. The drift catalog tracks discrepancies and triggers regeneration when >20% stale.

8. **Both wrapper formats supported.** Flat .md files (Dr Adelaida) and directory/SKILL.md (Dr Adelaida skills). Bootstrap detects which format the target tool expects.

9. **Lifecycle for everything.** Skills, commands, state files all have a lifecycle. Nothing is static.

10. **Spec-as-Truth generalized.** The mockup-as-truth pattern (TRHM) generalizes to any upstream contract — API spec, design mockup, schema.

11. **Brownfield extraction is a pattern.** Extracting projects from siblings (Mwanza from Bisou Bailey) is a re-usable workflow, not a one-off script.

12. **Phase classification is algorithmic, not hardcoded.** The 5-phase (Foundation → Infrastructure → Core Business → Content → Frontend) is one possible output of the classifier, not the classifier itself.

13. **npm package, not cloned repo.** The tool is distributed as an npm package — `npm install -D skills-commands`, updated via `npm update`. Generated artifacts are project-specific output, version-controlled in the project repo. This follows the 2026 standard: ESLint, Prettier, Biome, Knip — every successful dev tool separates the TOOL from the OUTPUT. The MCP runtime layer is deferred to v2 because the reference projects prove it's not needed for core value.

14. **Single-file skills for v1, directory format for v2.** Generated skills use single `.md` files initially — matching the proven pattern from all four reference projects. The directory format (SKILL.md + workflows/ + references/) enables progressive disclosure and reduces token cost 60-80% for complex skills, but is a v2 optimization. Single files work now.

15. **Markdown MCP routing for v1, composable pipelines for v3.** Generated skills include natural-language MCP routing tables in markdown with fallback instructions — the same format that works in Dr Adelaida and Bisou Bailey. Executable pipeline composition with MCPC interop is deferred until the generation engine is proven and the MCP protocol matures (5 protocol limitations identified).
